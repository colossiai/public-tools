# 为什么 0/1 背包不能直接定义 `dp[容量] = 最大总价值`？

先说结论：**其实可以只用 `dp[c] = 最大总价值`**——`08_knapsack_01.py` 末尾（94-102 行）的"空间优化"版本就是这么做的。但这里有一个**魔鬼细节**：一维之所以能成立，靠的是**倒序遍历容量**这个技巧，否则它会悄悄变成完全背包（每件物品能拿无数次），答案就错了。

真正的问题其实是：**为什么最"自然"的那种一维定义会出问题？**

---

## 一、为什么"直觉版"一维会错

假设你天真地写：

```python
dp = [0] * (capacity + 1)
for i in range(1, n + 1):
    w, v = weights[i-1], values[i-1]
    for c in range(w, capacity + 1):   # ⚠️ 正序
        dp[c] = max(dp[c], dp[c - w] + v)
```

问题出在这一行：`dp[c - w]`。

在 0/1 背包里，"拿第 i 件"要求剩下的容量 `c - w` 交给**前 i-1 件**去分配（即上一轮 `i-1` 的值）。但正序遍历时，`dp[c - w]` 在本轮**已经被更新过了**——它可能已经包含了"第 i 件"自己。于是第 i 件就被拿了不止一次，变成了**完全背包**。

用文件里的物品举个具体例子。物品 A：`w=1, v=1`，容量 7。正序遍历 A 这一轮：

```
c=1: dp[1] = max(dp[1], dp[0]+1) = 1   # 拿了 1 个 A
c=2: dp[2] = max(dp[2], dp[1]+1) = 2   # dp[1] 已是"拿过 A"的值 → 又拿一个 A！
c=3: dp[3] = max(dp[3], dp[2]+1) = 3   # 拿了 3 个 A ...
```

A 只有一件，却被拿了 7 次。**这就是为什么单纯的 `dp[c]` 一维定义"看起来能行，实则会错"。**

---

## 二、根本原因：状态的"无后效性"需要两个维度

`dp[c]` 这个定义本身**信息不足**。转移时你需要区分两件事：

| 你需要的 | 一维 `dp[c]` 能提供的 |
|---|---|
| "还没考虑第 i 件时，容量 c 的最优值" | ❌ 分不清 |
| "已经考虑完第 i 件后，容量 c 的最优值" | ❌ 分不清 |

同一个 `dp[c]` 在遍历过程中，**一半是旧值一半是新值**，你没法判断它属于哪个阶段。而 0/1 背包的正确性恰恰依赖"第 i 件只能基于**前 i-1 件**的结果来决策"——这就是**多阶段决策**的无后效性要求。

二维 `dp[i][c]` 通过多出来的 `i` 维度，把"第几阶段"显式记下来，`dp[i-1][...]` 永远指向上一阶段的干净值，天然正确（这正是文件 48 行的定义）。

---

## 三、那一维为什么最后又能救回来？

**倒序遍历**：

```python
for c in range(capacity, w - 1, -1):   # 从大到小
    dp[c] = max(dp[c], dp[c - w] + v)
```

倒序时，更新 `dp[c]` 用到的 `dp[c - w]`（下标更小）**本轮还没被碰过**，仍是上一轮 `i-1` 的旧值——等价于二维里的 `dp[i-1][c-w]`。这样每件物品就只会被拿一次。

所以严格说：

- **可以**用 `dp[c] = 最大总价值`；
- 但它是二维的**空间压缩**，正确性建立在"倒序"上；
- 教学时先写二维 `dp[i][c]`，是因为**它的定义自带阶段信息，不需要任何遍历顺序的技巧就正确**，更容易理解和证明。

---

## 一句话总结

**不是不能只用容量做状态，而是"只用容量"会丢掉"考虑到第几件"这个阶段信息；二维显式保留它，一维靠倒序隐式保留它。**

> 倒序 = 0/1 背包（每件最多一次），正序 = 完全背包（每件无限次）。
