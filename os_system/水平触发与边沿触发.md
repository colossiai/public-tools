**水平触发（Level Triggered, LT）** 和 **边沿触发（Edge Triggered, ET）** 是操作系统中 I/O 多路复用（尤其是 `epoll`）最重要的两个概念，它们决定了**什么时候通知程序某个文件描述符（fd）可以读/写**。

可以把它们想象成门铃。

---

# 一个生活例子

假设有人送快递。

## 水平触发（LT）

快递一直放在门口。

只要快递还没拿走，门铃就会一直响。

```
快递到了
      ↓
🔔🔔🔔🔔🔔🔔🔔🔔
直到你拿走
```

程序也是一样。

只要 socket 里还有数据没有读完：

```
socket:
---------------------
abcde
---------------------
```

即使你只读了：

```
read -> abc
```

还剩：

```
de
```

下一次 `epoll_wait()`：

```
还有数据！
```

仍然通知你。

因此：

> **只要条件满足，就一直通知。**

---

# 边沿触发（ET）

门铃只响一次。

```
快递到了
    ↓
🔔
```

如果你没去拿：

```
快递还在门口
```

不会再响。

程序也是一样。

socket：

```
空
```

后来收到：

```
ABCDE
```

状态变化：

```
空
↓

有数据
```

发生了一次"边沿"（状态变化）。

于是：

```
epoll_wait()

返回一次
```

如果你只读：

```
ABC
```

剩：

```
DE
```

由于 socket 仍然是：

```
有数据
```

没有再次发生：

```
空 -> 有数据
```

所以：

```
epoll_wait()

不会再通知！！
```

直到：

```
DE读完

socket变空

↓

又收到FG

空 -> 有数据
```

再次发生边沿：

```
再通知一次
```

---

# 为什么叫 Edge？

数字电路里面有：

```
0 -> 1
```

叫：

**Rising Edge**

```
1 -> 0
```

叫：

**Falling Edge**

ET 的意思就是：

> **只有状态发生变化才通知。**

---

# 为什么叫 Level？

因为一直保持在某个"电平"。

例如：

```
socket一直有数据
```

条件一直成立。

那么：

```
每次epoll_wait()

都会返回
```

这就是：

**Level Triggered**

---

# 一个时间线

假设：

```
socket收到：

ABCDE
```

程序每次只读 2 字节。

## LT

第一次：

```
ABCDE

epoll返回
```

读：

```
AB
```

剩：

```
CDE
```

下一次：

```
epoll_wait()

还有数据

继续返回
```

读：

```
CD
```

剩：

```
E
```

下一次：

```
继续返回
```

直到：

```
空
```

---

## ET

第一次：

```
ABCDE

状态：

空 -> 有数据

epoll返回
```

读：

```
AB
```

剩：

```
CDE
```

下一次：

```
epoll_wait()

什么也没有！！
```

因为：

```
socket一直都是：

有数据
```

没有发生：

```
空 -> 有数据
```

---

# ET 为什么必须循环读取？

因此 ET 有一个经典写法：

```c
while (true) {
    n = read(fd, buf, sizeof(buf));

    if (n > 0) {
        // 处理数据
    } else if (n == -1 && errno == EAGAIN) {
        // 数据读完了
        break;
    } else {
        // 出错或连接关闭
        break;
    }
}
```

为什么？

因为：

> **ET 通知通常只有一次，必须一次把缓冲区的数据全部读空，直到 `read()` 返回 `EAGAIN`（或 `EWOULDBLOCK`）。**

因此 ET **必须配合非阻塞 socket**。

---

# LT 可以不用这样吗？

可以。

LT 即使：

```c
read(fd, buf, 100);
```

只读一点点。

下次：

```
epoll_wait()

还会通知
```

所以比较容易写。

---

# 性能区别

假设 socket 中有 100MB 数据。

程序一次只读 1KB。

## LT

每次：

```
epoll_wait()

↓

read 1KB

↓

epoll_wait()

↓

read 1KB
```

系统需要不断把同一个 fd 放回就绪队列。

通知次数很多。

---

## ET

```
epoll_wait()

↓

一次通知

↓

while(read())

↓

一直读到EAGAIN
```

只通知一次。

因此：

* 更少的系统调用（`epoll_wait` 唤醒次数更少）
* 更少的事件分发
* 在高并发场景下通常性能更好

---

# 为什么很多高性能服务器喜欢 ET？

例如：

* Nginx
* Redis
* Netty（Linux epoll）
* Muduo

都会优先使用 ET，因为：

* 通知次数更少
* 减少 `epoll_wait()` 唤醒
* 降低内核与用户态切换开销
* 更适合事件驱动架构

代价是：

* 代码更复杂
* 必须使用非阻塞 I/O
* 必须一直读（或写）到返回 `EAGAIN`
* 否则容易出现“数据还在缓冲区，但再也收不到事件”的问题

---

# 一张总结表

| 特性              | 水平触发（LT）  | 边沿触发（ET） |
| --------------- | --------- | -------- |
| 通知时机            | 条件满足就一直通知 | 仅状态变化时通知 |
| 未读数据            | 会反复通知     | 不会再次通知   |
| 编程难度            | 简单        | 较复杂      |
| 是否必须非阻塞         | 推荐，但不是必须  | **必须**   |
| 是否必须读到 `EAGAIN` | 不必须       | **必须**   |
| 系统调用次数          | 较多        | 较少       |
| 高并发性能           | 较好        | 通常更好     |

## 一句话记忆

* **水平触发（LT）**：**"还有数据，我一直提醒你。"**
* **边沿触发（ET）**：**"数据刚到了，我只提醒你一次，剩下的靠你自己一次性处理完。"**
