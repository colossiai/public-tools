# #map c++ condition variable to assembly

Mapping a **C++ `std::condition_variable`** to assembly requires understanding that it is **not a single instruction**. It is a **library abstraction built on OS synchronization primitives**, which themselves ultimately rely on **CPU atomic instructions + kernel scheduling**.

I'll walk from **C++ → libc → OS → CPU assembly**.

---

# 1. C++ Level

Typical usage:

```cpp
std::mutex m;
std::condition_variable cv;
bool ready = false;

void worker() {
    std::unique_lock<std::mutex> lock(m);
    cv.wait(lock, []{ return ready; });
}

void signal() {
    std::lock_guard<std::mutex> lock(m);
    ready = true;
    cv.notify_one();
}
```

`cv.wait(lock)` does **three critical operations**:

1. **Atomically unlock mutex**
2. **Sleep the thread**
3. **Re-lock mutex after wake**

This atomicity is the core difficulty.

---

# 2. What the Standard Library Actually Calls

On **Linux**, `std::condition_variable` usually maps to:

```
pthread_cond_wait
pthread_cond_signal
```

From **POSIX threads.

---

# 3. pthread_cond_wait Implementation

Inside **glibc**, `pthread_cond_wait` eventually uses:

```
futex()
```

The **futex** syscall.

Conceptually:

```
unlock(mutex)
futex_wait(cond_var)
lock(mutex)
```

But the unlock + sleep is made **atomic via futex protocol**.

---

# 4. Futex Assembly-Level Behavior

User-space first tries **atomic instructions** before entering the kernel.

Typical x86 sequence:

### Load condition variable

```asm
mov eax, [cond_var]
cmp eax, expected
jne skip_sleep
```

### Futex syscall

```asm
mov rax, SYS_futex
mov rdi, cond_var
mov rsi, FUTEX_WAIT
mov rdx, expected
syscall
```

Thread sleeps in kernel queue.

---

# 5. notify_one() path

C++:

```cpp
cv.notify_one();
```

glibc:

```
pthread_cond_signal
```

Kernel wake:

```asm
mov rax, SYS_futex
mov rdi, cond_var
mov rsi, FUTEX_WAKE
mov rdx, 1
syscall
```

Kernel wakes **one sleeping thread**.

---

# 6. CPU Atomic Instructions Used

The mutex protecting the condition variable uses **atomic instructions** such as:

### x86

```
LOCK CMPXCHG
LOCK XADD
LOCK BTS
```

Example spin-lock acquire:

```asm
mov eax, 0
mov ecx, 1
lock cmpxchg [mutex], ecx
jne retry
```

These enforce **cache coherency + memory ordering**.

---

# 7. Full Timeline

Thread A waits:

```
lock(mutex)
check condition
unlock(mutex)
futex_wait(cond)
sleep
```

Thread B signals:

```
lock(mutex)
update condition
unlock(mutex)
futex_wake(cond)
```

Kernel wakes thread A → it re-locks mutex.

---

# 8. Conceptual Assembly Flow

Simplified:

### wait()

```asm
; unlock mutex
mov [mutex], 0

; sleep
mov rax, SYS_futex
mov rdi, cond
mov rsi, FUTEX_WAIT
syscall
```

### notify_one()

```asm
mov rax, SYS_futex
mov rdi, cond
mov rsi, FUTEX_WAKE
mov rdx, 1
syscall
```

---

# 9. Why Condition Variables Exist

They avoid **busy spin loops**.

Without condition variable:

```asm
spin:
mov eax, [ready]
test eax, eax
jz spin
```

This burns CPU.

Condition variables allow **kernel scheduling sleep**.

---

# 10. Real Cost

Latency tiers:

| Mechanism               | Latency   |
| ----------------------- | --------- |
| atomic spin             | ~10–50 ns |
| futex uncontended       | ~100 ns   |
| futex kernel sleep/wake | ~1–5 µs   |

That's why **low latency systems (HFT)** avoid condition variables.

They prefer:

* spinlocks
* lock-free queues
* cache-line polling
